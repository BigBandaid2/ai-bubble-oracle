# AI Bubble Oracle

**Live: https://ai-bubble-oracle.vercel.app**

Unofficial monitoring oracle for the Polymarket market **"AI bubble burst in 2026?"**
(event `ai-bubble-burst-by`, condition id `0x857398c4502bc725fef7efb3cd503a30d3e18e486ab8173fdf505a76cf83b168`).

The market resolves YES if **at least 3 of 6** conditions occur within a 90-day
window. This tool tracks the conditions automatically from public data and shows,
in real time, where we stand against them — with every data source, processing
step, and rules ambiguity made inspectable. See the in-app **About** page for the
project's goals.

> This is a *monitoring* tool. The market itself resolves through UMA's
> optimistic oracle — humans reading prose — not through this or any code.

## Live site & automation

The site is three static pages — `dashboard.html`, `datasources.html`,
`about.html` — generated by Python and served on Vercel.

- **Daily data refresh:** a GitHub Action (`.github/workflows/daily-update.yml`)
  runs `python main.py update` every day at 22:30 UTC, regenerates the pages, and
  commits `dashboard.html`, `datasources.html`, and `data/h100_history.csv`.
- **Auto-deploy:** each push to `main` triggers a Vercel redeploy (via Vercel's
  GitHub integration), so the daily commit publishes automatically.
- **Data persistence:** `oracle.db` is rebuilt from Yahoo on every run and is
  *not* committed (it's git-ignored). The only state that must persist between
  runs — the accumulating H100 rental proxy readings — lives in the small
  committed `data/h100_history.csv`.

No third-party Python dependencies (stdlib only); Vercel serves the pre-built
static HTML with no build step.

## Conditions covered by data

Four of the six conditions now resolve from automated data:

| Condition | Trigger | Source |
|---|---|---|
| `nvda_down_50` | NVDA down 50% from all-time high | Yahoo Finance |
| `soxx_down_40` | SOXX down 40% from all-time high | Yahoo Finance |
| `supplier_down_50` | Any of TSM, ASML, AVGO, ANET, SMCI down 50% from ATH | Yahoo Finance |
| `h100_rental_dollar` | H100 rental ≤ $1.00/hr for 5 straight days | Vast.ai proxy (see below) |
| `openai_bankruptcy` / `anthropic_bankruptcy` | Chapter 7/11 filing | CourtListener/RECAP daily scan |

The bankruptcy scan queries `caseName:(<entity>) AND chapter:(7 OR 11)` daily
(validated against FTX's 2022 Chapter 11 wave), applies an exact-substring name
gate against stemming noise, and records candidate counts to
`data/bankruptcy_history.csv`. A candidate **never** flips the condition by
itself — it counts only after human review. Only the OpenAI-acquisition
condition still awaits a data source (news monitoring).

**How you're notified of a candidate:** the daily Action opens a GitHub issue
titled "⚠️ Bankruptcy candidate pending review" (repo watchers get an
email/app notification; deduped while one is open). The dashboard card also
shows the candidate count in red and the buzz coefficient jumps to ≥ 0.90.

**How to confirm (the human gate):**
1. Review the docket on courtlistener.com — is it a genuine parent-company
   Chapter 7/11 petition (not a subsidiary, adversary proceeding, or name
   collision)?
2. If genuine: add it to `CONFIRMED_BANKRUPTCIES` in `oracle/config.py`, e.g.
   `{"OpenAI": "2026-08-14"}` (the filing date it counts from), and commit.
3. Run the **Daily data update** workflow (Actions tab → Run workflow) — it
   regenerates the site with the condition MET (buzz = 1.00) and the push
   auto-deploys.
4. False positive? Just close the issue; the scan keeps running daily.

- Data: Yahoo Finance public chart API (split-adjusted daily OHLC) + Vast.ai
  public bundles API — neither needs a key
- Storage: `oracle.db` (SQLite, created on first run)
- No third-party Python dependencies (stdlib only, Python 3.10+)

### H100 rental price (condition 5)

The rules resolve on the **SiliconData Silicon Index**, published only via
Bloomberg / enterprise API — no public feed. That authoritative source is left
as a **future plan**. For now the oracle tracks a free public **proxy** into
`h100_prices` (keyed by date + index tier):

- **Neocloud (SDH100RT)** — the spot-like marketplace tier (~$2/hr). Live: the
  median $/hr of verified on-demand H100 SXM offers on the **Vast.ai**
  marketplace, collected daily. Tracks the published index closely (~$2.2 vs.
  the ~$2.16 SDH100RT in SiliconData's Sept-2025 post).
- **Hyperscaler** — the reserved/committed tier (AWS/GCP/Azure/Oracle), ~3×
  pricier (~$7.5/hr). No free live feed, so seeded from values published in
  SiliconData's April-2026 post.

The **neocloud/hyperscaler ambiguity** is exposed as a dashboard toggle: the
rules don't say which index, and the $1.00 trigger only plausibly refers to
neocloud (a ~50% fall) rather than hyperscaler (near-impossible). A few real
published values seed each tier for historical context. The condition is met
when the daily reading is ≤ $1.00 for 5 consecutive days. It's a proxy / leading
indicator, not the resolution source; because Vast.ai exposes only current
offers, history accumulates going forward.

### Usage

```
python main.py update       # fetch prices (full history on first run, 90d after), rebuild events + both HTML pages
python main.py status       # condition-tree report: drawdowns, distance-to-trigger, met/not-met
python main.py events       # every historical threshold crossing with dates
python main.py html         # regenerate dashboard.html + datasources.html without refetching
python main.py datasources  # regenerate only datasources.html
```

### Dashboard

`update` (or `html`) writes a self-contained `dashboard.html` — open it in any
browser, no server or internet needed. It renders the full condition tree:

- one chart per condition: price + running ATH + trigger line for data-driven
  leaves, a children-met count chart (with the required-count line) for parent
  conditions, and a placeholder for conditions with no feed yet (Phase 3/4)
- red shading marks periods when a condition was met; a lighter shade marks the
  "trailing" period when a recovered condition still counts (see window options)
- right sidebar: nested, expandable condition list with indicators and "last
  met" dates; click any condition to jump to its chart

### Interactive options (resolution ambiguities)

The market's rules are ambiguous in ways that change resolution, so the sidebar
exposes them as live toggles. All interpretation happens in the browser — the
payload ships raw daily data and the whole condition tree recomputes on toggle,
no re-fetch. Each chart shows chips for the options currently affecting it.

**All-time-high basis** — "all-time high" isn't defined as close or intraday:
- *End-of-day*: closing price vs. highest close (default)
- *Intraday*: day's low vs. highest intraday high (triggers more easily)

**90-day window** — "resolve once 3 conditions have been met within 90 days" has
two defensible readings:
- *Trailing 90 days* (default): a condition stays "met" for 90 days after it
  crosses, so the market resolves YES if 3 conditions cross within any 90-day
  span. Equivalent to "3 trigger-events within a 90-day window."
- *Instant*: 3 conditions must be past threshold on the same day.

Indicator dots: solid red = past threshold now; hollow red = met via the
trailing window (crossed recently, since recovered); green = not met; hollow
gray = no data feed. Switching to *Instant* removes hollow states; switching
basis/window visibly changes the count charts and "last met" dates (e.g. the
root's last-met is 2022-11-09 under Instant vs. 2023-02-07 under Trailing).

The contract is modeled as an arbitrary-depth condition tree in
`oracle/config.py` (`CONTRACT`): `drawdown` leaves resolve from price data,
`count` parents resolve from how many children are met, `manual` leaves await
a data feed. Adding a condition = adding a node there.

### Data Sources page

`datasources.html` (linked from the dashboard nav) is a master-detail explorer
of the ETL pipeline. It opens on a **Data Source Intro** landing page with a
card per source. Each source has its own **overview & lineage** page — a short
intro, an **options / ambiguities / caveats** section (e.g. the H100
neocloud-vs-hyperscaler tier, the "H100 is an approximation" caveat), its own
top-to-bottom **asset graph**, and a **cardinality** table whose rows line up
one-to-one with the graph's assets (so you can see where the pipeline fans out
and collapses). Each source runs its own pipeline to conclusion through its own
**mart** and **serve** steps (e.g. the equity leaves end in `fct_condition_daily`
→ `fct_contract_resolution` → `exposure.dashboard`; H100 ends in its own
`fct_h100_rental` → `exposure.h100_series` with rental-specific columns). The
browser combines the served slices into the 3-of-6 contract count.

The right sidebar (on the right) lists each source as an **accordion** group —
one expanded at a time — and its ordered **assets** (Dagster's term for a
materialized table/view handed from one step to the next). The **option toggles**
live in a slide-out **drawer** (the ⚙ Options button), and drive the reactive
worked examples. Both pages use an **app-shell** layout (content and sidebar
scroll in their own panes, so their scrollbars never sit adjacent). Selecting an
asset shows it in full:

- lineage (clickable upstream inputs → this asset → downstream consumers)
- schema, with every column tagged by origin — key / passthrough / cast /
  derived / seed / meta — so you can see what each step adds vs. carries through
- a **real example row** — one condition (SMCI) traced through every stage with
  actual numbers from `oracle.db`, so the shape and state of the data at each
  step is concrete
- the transforms applied (named after their dbt / dbt-utils equivalents), a SQL
  sketch, tests, materialization, refresh cadence, and cardinality

The Overview page shows the asset graph, a column-origin legend, and how row
counts fan out and roll up. Each transform is expressed in standard terms
(`dbt_utils.date_spine`, `deduplicate`, `generate_surrogate_key`, window
functions, recursive-CTE rollup, etc.). Yahoo Finance is the only live source;
the H100 index, court dockets, and news feeds are documented as planned
(Phase 3/4). The example numbers are computed in `oracle/datasources.py`.

## Rules ambiguities this tool tracks both sides of

1. **Intraday vs. closing all-time high.** The rules just say "all-time high".
   We compute both bases: `close` (close vs. max close) and `intraday`
   (day low vs. max intraday high). Where they disagree, the report shows both.
2. **Pre-existing drawdowns.** SMCI has been more than 50% below its 2024 ATH
   since long before this market opened, yet earlier deadlines resolved NO —
   resolvers evidently require the *event* to occur in the window, not the
   *state* to exist. That's why we store dated crossing events, not just
   current status. The 90-day clustering logic over these events is Phase 2.
3. **Split/dividend adjustment.** We use split-adjusted quoted prices
   (Yahoo `close`), not dividend-adjusted (`adjclose`), matching how prices
   are quoted publicly.
